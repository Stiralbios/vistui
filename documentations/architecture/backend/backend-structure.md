# Vistui Backend Structure

The backend follows a **feature-based** organization pattern. Each feature is a self-contained directory under `sources/backend/<feature>/`.

## Feature Directory Pattern

Every feature follows the same file structure:

```
sources/backend/<feature>/
├── models.py       # SQLAlchemy ORM models (Data Objects)
├── schemas.py      # Pydantic models (validation/serialization)
├── stores.py       # Database CRUD operations
├── services.py     # Business logic layer
├── apis.py         # FastAPI routers and endpoints
├── dependencies.py # FastAPI dependencies
└── constants.py    # Enums, constants
```

## File Responsibilities

### models.py

SQLAlchemy declarative models mapped to PostgreSQL tables.

Example from `message/models.py`:

```python
class MessageDO(Base):
    __tablename__ = "message"

    id: Mapped[uuid.UUID] = mapped_column(Uuid, primary_key=True, index=True)
    chat_id: Mapped[uuid.UUID] = mapped_column(ForeignKey("chat.id"))
    prev_message_id: Mapped[uuid.UUID | None] = mapped_column(ForeignKey("message.id"), nullable=True)
    next_message_id: Mapped[uuid.UUID | None] = mapped_column(ForeignKey("message.id"), nullable=True)
    full_text: Mapped[str] = mapped_column(Text)
    search_vector: Mapped[Any] = mapped_column(TSVectorType())
    embedding: Mapped[Vector] = mapped_column(Vector(768), nullable=True)
    role: Mapped[str] = mapped_column(String(20))
    salience_score: Mapped[float | None] = mapped_column(Float, nullable=True)
    processing_state: Mapped[dict] = mapped_column(JSONB)
```

Rules:

- Table names are lowercase singular (e.g., `message`, not `messages`).
- Primary keys are `id` columns of UUID type.
- Message IDs are caller-provided; all other primary keys are generated by the database.
- Relationships use `back_populates` for bidirectional access.

### schemas.py

Pydantic models for request/response validation.

Schema naming convention:

- `FeatureBase` — shared fields.
- `FeatureRead` — full representation returned to clients.
- `FeatureInternal` — internal representation used between API and Service.
- `FeatureCreate` / `FeatureCreateInternal` — creation input.
- `FeatureUpdate` / `FeatureUpdateInternal` — update input.
- `FeatureFilter` — query filter parameters.
- `MemoryResponse` / `MemoryItem` — retrieval-specific response shapes.

Example:

```python
class MessageRead(MessageBase):
    id: uuid.UUID
    chat_id: uuid.UUID
    prev_message_id: uuid.UUID | None
    next_message_id: uuid.UUID | None
    role: MessageRole
    keywords: list[str]
    salience_score: float | None
    processing_state: ProcessingState
    created_at: datetime
    updated_at: datetime

class MessageCreateInternal(MessageCreate):
    chat_id: uuid.UUID  # injected by API layer
```

### stores.py

Database operations with `with_async_session` decorator for automatic session management.

```python
class MessageStore:
    @staticmethod
    @with_async_session
    async def create(session: AsyncSession, message: MessageCreateInternal) -> MessageInternal:
        orm_object = MessageDO(**message.model_dump(exclude={"context"}))
        session.add(orm_object)
        await session.flush()
        return MessageInternal.model_validate(orm_object)
```

Rules:

- Stores are stateless (all methods are `@staticmethod`).
- One store writes to **one model only**.
- Complex reads across models belong in specialized **QueryStore** classes.
- Use `@with_async_session` for automatic transaction handling.

### services.py

Business logic layer. Services may call multiple stores.

Rules:

- Services **may NOT call other services**.
- Shared business logic is extracted to standalone business functions that call stores.
- Services validate permissions and state transitions.
- Services return internal schemas.

Example:

```python
class MessageService:
    async def save(self, message: MessageCreateInternal) -> MessageInternal:
        await self._validate_linked_list(message)
        if message.context.retrieve:
            message = await self._compute_embedding_and_keywords(message)
        return await self.store.upsert(message)
```

### apis.py

FastAPI routers with endpoints.

Rules:

- Receive **external schemas** from clients.
- Return **external schemas** (`Read`, `Create`) to clients.
- Convert external to **internal schemas** before calling services.
- Inject `chat_id`, `user_id`, and `context` in the API layer.

Example:

```python
@router.put("/{message_id}", response_model=MessageSaveResponse)
async def save_message(
    chat_id: uuid.UUID,
    message_id: uuid.UUID,
    message: MessageCreate,
    retrieve: bool = False,
) -> MessageSaveResponse:
    message_internal = MessageCreateInternal.model_validate(
        {**message.model_dump(exclude_unset=True), "chat_id": chat_id, "id": message_id}
    )
    message_internal.context = MessageSaveContext(retrieve=retrieve)
    return await MessageService().save(message_internal)
```

### dependencies.py

FastAPI dependencies reused across endpoints.

In v1, authentication is minimal: every mutating endpoint requires a valid user, but all authenticated users share the same rights. See [Authentication](auth.md).

### constants.py

Enums and constants.

Example:

```python
class MessageRole(StrEnum):
    SYSTEM = "system"
    USER = "user"
    ASSISTANT = "assistant"

class ProcessingStatus(StrEnum):
    WAITING = "waiting"
    PROCESSING = "processing"
    DONE = "done"
    FAILED = "failed"
```

## Feature List

| Feature | Status | Description |
|---------|--------|-------------|
| `user` | Planned | User management and authentication. |
| `chatgroup` | Planned | Configuration, prompts, ranking formula, topics, and facts. |
| `chat` | Planned | Chat windows bound to exactly one ChatGroup. |
| `message` | Planned | Linked-list messages with embedding, keywords, and salience. |
| `event` | Planned | Contiguous message summaries within a Chat. |
| `topic` | Planned | Recurring theme summaries across a ChatGroup. |
| `fact` | Planned | Persistent facts across a ChatGroup. |
| `retrieval` | Planned | Vector + BM25 search, ranking, and token-budget packing. |
| `batch` | Planned | APScheduler worker for embeddings, salience, and consolidation. |
| `health` | Planned | `GET /health` endpoint. |

## Test Structure

Tests mirror the backend structure:

```
tests/backend/
├── conftest.py          # Test database setup, fixtures
├── config.py            # Test config
├── <feature>/
│   ├── factories.py     # polyfactory factories for model creation
│   └── test_apis.py     # API integration tests
```

Each test file has an equivalent in `sources/backend/<feature>/`.
