# Vistui API Contract

## Context

This document defines the external REST API for Vistui. It is intentionally frontend-agnostic: the callers are chatbot frontends, extensions, or MCP servers, not human users directly.

The API follows resource-oriented design using UUID identifiers. Message IDs are caller-provided to support idempotent creation and editing via `PUT`.

## Base URL and versioning

All endpoints are prefixed with:

```
/api/v1
```

Example:

```
POST /api/v1/users
PUT /api/v1/chats/{chat_id}/messages/{message_id}
```

No versioning in media type; version is in the path.

## Common conventions

### Content type

Requests and responses use `application/json` unless otherwise noted.

### Timestamps

All timestamps are ISO 8601 strings in UTC (e.g., `2026-07-11T12:00:00Z`).

### UUIDs

All resource identifiers are UUID v4 strings. Message IDs are generated by the caller; all other IDs are generated by the server.

### Pagination

List endpoints use cursor-based pagination with:

| Query param | Description |
|-------------|-------------|
| `limit` | Max items to return (default 20, max 100). |
| `cursor` | Opaque cursor from the previous response. |

Response shape:

```json
{
  "items": [...],
  "next_cursor": "opaque-string-or-null",
  "has_more": true
}
```

### Errors

Errors follow RFC 7807 `application/problem+json`:

```json
{
  "type": "https://vistui.example/errors/not-found",
  "title": "Resource not found",
  "status": 404,
  "detail": "Chat with id ... does not exist."
}
```

Common status codes:

| Status | Meaning |
|--------|---------|
| 400 | Bad request (validation error). |
| 404 | Resource not found. |
| 409 | Conflict (e.g., duplicate UUID, inconsistent linked list). |
| 422 | Unprocessable entity (semantic violation). |
| 500 | Internal server error. |

## Endpoints

### Users

#### Create user

```
POST /users
```

Request:

```json
{
  "name": "Alice",
  "email": "alice@example.com"
}
```

Response: `201 Created`

```json
{
  "id": "uuid",
  "name": "Alice",
  "email": "alice@example.com",
  "created_at": "2026-07-11T12:00:00Z"
}
```

#### Get user

```
GET /users/{user_id}
```

#### Update user

```
PATCH /users/{user_id}
```

### ChatGroups

#### Create ChatGroup

```
POST /chatgroups
```

Request:

```json
{
  "user_id": "uuid",
  "name": "Companion Chats",
  "description": "Shared memory for companion roleplay.",
  "config": {
    "token_budget": 2000,
    "memory_ratio": {
      "message": 0.40,
      "event": 0.10,
      "topic": 0.30,
      "fact": 0.20
    },
    "use_llm_memory_ratio": true,
    "protected_message_count": 10,
    "topic_update_threshold": 0.75,
    "fact_update_threshold": 0.80,
    "fact_history_count": 5,
    "max_facts_per_message": 5,
    "max_topics_per_message": 5,
    "batch_inactivity_seconds": 3600,
    "processing_timeout_seconds": 3600
  },
  "prompts": {
    "salience": "...",
    "keywords": "...",
    "memory_ratio": "...",
    "event": "...",
    "topic_candidate": "...",
    "topic_update": "...",
    "fact_candidate": "...",
    "fact_update": "..."
  },
  "ranking_formula": "lambda vector_score, bm25_score, salience, linked_count: max(vector_score * salience * linked_count, bm25_score * salience * linked_count)"
}
```

Omitted `config`, `prompts`, and `ranking_formula` are filled from system defaults.

Response: `201 Created`

```json
{
  "id": "uuid",
  "user_id": "uuid",
  "name": "Companion Chats",
  "description": "Shared memory for companion roleplay.",
  "config": { ... },
  "prompts": { ... },
  "ranking_formula": "...",
  "created_at": "2026-07-11T12:00:00Z",
  "updated_at": "2026-07-11T12:00:00Z"
}
```

#### Get ChatGroup

```
GET /chatgroups/{chatgroup_id}
```

#### Update ChatGroup

```
PATCH /chatgroups/{chatgroup_id}
```

Allows updating `name`, `description`, `config`, `prompts`, and `ranking_formula`. Changes apply to future operations only; existing memory is not retroactively re-ranked or re-consolidated.

#### Delete ChatGroup

```
DELETE /chatgroups/{chatgroup_id}
```

Cascades to Chats, Messages, Events, Topics, and Facts.

### Chats

#### Create Chat

```
POST /chats
```

Request:

```json
{
  "chatgroup_id": "uuid-or-null",
  "name": "Date with Aria",
  "description": "A romantic roleplay scenario."
}
```

If `chatgroup_id` is null, the API creates a new ChatGroup using the provided `name` and `description` and assigns the Chat to it.

Response: `201 Created`

```json
{
  "id": "uuid",
  "chatgroup_id": "uuid",
  "name": "Date with Aria",
  "description": "A romantic roleplay scenario.",
  "created_at": "2026-07-11T12:00:00Z",
  "updated_at": "2026-07-11T12:00:00Z"
}
```

#### Get Chat

```
GET /chats/{chat_id}
```

#### List Chats in ChatGroup

```
GET /chatgroups/{chatgroup_id}/chats
```

#### Update Chat

```
PATCH /chats/{chat_id}
```

Only `name` and `description` may be updated. `chatgroup_id` is immutable.

#### Delete Chat

```
DELETE /chats/{chat_id}
```

Cascades to Messages, Events, and join links. Topics and Facts in the parent ChatGroup remain unless no other Chat references them via links.

### Messages

#### Save or edit message

```
PUT /chats/{chat_id}/messages/{message_id}
```

`message_id` is caller-provided. This endpoint handles both creation and idempotent overwrite.

Query parameters:

| Param | Type | Default | Description |
|-------|------|---------|-------------|
| `retrieve` | boolean | `false` | If `true`, compute and return relevant memory for this message. |

Request:

```json
{
  "prev_message_id": "uuid-or-null",
  "next_message_id": "uuid-or-null",
  "full_text": "Aria, do you remember our first date?",
  "role": "user"
}
```

Rules:

- If `message_id` does not exist, a new Message is created.
- If `message_id` exists, it is overwritten in place.
- The linked list is validated: `prev_message_id` and `next_message_id` must be consistent with existing neighbors.
- The embedding and keywords are recomputed.
- Memory ratio is determined per retrieval request, either by LLM (if `use_llm_memory_ratio` is enabled) or from the ChatGroup `memory_ratio` config.
- Retrieval is performed only if `retrieve=true`.
- Batch processing state is reset to `waiting` for embedding/salience on every save, and for event/topic/fact only if the message is outside the protected window.

Response: `200 OK` (or `201 Created` on first creation)

```json
{
  "id": "uuid",
  "chat_id": "uuid",
  "prev_message_id": "uuid-or-null",
  "next_message_id": "uuid-or-null",
  "full_text": "Aria, do you remember our first date?",
  "role": "user",
  "keywords": ["first date", "Aria", "memory"],
  "salience_score": 0.82,
  "processing_state": {
    "embedding": "waiting",
    "salience": "waiting",
    "event": "waiting",
    "topic": "waiting",
    "fact": "waiting"
  },
  "created_at": "2026-07-11T12:00:00Z",
  "updated_at": "2026-07-11T12:00:01Z",
  "memory": {
    "token_budget": 2000,
    "memory_ratio": {
      "message": 0.40,
      "event": 0.10,
      "topic": 0.30,
      "fact": 0.20
    },
    "used_tokens": 1234,
    "items": [
      {
        "type": "fact",
        "id": "uuid",
        "full_text": "User and Aria went on a picnic date last spring.",
        "relevance": 0.91,
        "estimated_tokens": 12
      },
      {
        "type": "event",
        "id": "uuid",
        "full_text": "Aria and user discussed past dates and shared memories.",
        "relevance": 0.78,
        "estimated_tokens": 18
      }
    ]
  }
}
```

If `retrieve=false`, the `memory` field is omitted.

#### Get message

```
GET /chats/{chat_id}/messages/{message_id}
```

#### Delete message

```
DELETE /chats/{chat_id}/messages/{message_id}
```

Deletes the message and repairs the linked list. Does not cascade to Events, Topics, or Facts; links become dangling and are ignored by future processing.

#### List messages in Chat

```
GET /chats/{chat_id}/messages
```

Returns messages in linked-list order. Supports cursor pagination.

### Batch operations (admin / debug)

#### Trigger batch worker manually

```
POST /admin/batch/trigger
```

Immediate no-op if worker is already running. Used mainly for testing.

#### Get batch status

```
GET /admin/batch/status
```

Response:

```json
{
  "running": true,
  "current_chatgroup_id": "uuid-or-null",
  "current_pipeline": "fact",
  "queue_depth": 42
}
```

## Memory item response shape

A memory item returned by retrieval has the following shape regardless of underlying type:

```json
{
  "type": "message | event | topic | fact",
  "id": "uuid",
  "full_text": "string",
  "relevance": 0.85,
  "estimated_tokens": 15,
  "metadata": {
    "chat_id": "uuid-or-null",
    "chatgroup_id": "uuid-or-null",
    "role": "user-or-assistant-or-system-null",
    "start_at": "2026-07-11T12:00:00Z-or-null",
    "end_at": "2026-07-11T12:00:00Z-or-null"
  }
}
```

Token estimates use a simple tokenizer heuristic; exactness is not critical because callers may truncate or re-tokenize.

## Constraints and validation rules

1. `role` must be `system`, `user`, or `assistant`.
2. `full_text` must be non-empty and within a reasonable max length (TBD, suggested 32,000 characters).
3. `prev_message_id` and `next_message_id` must refer to messages in the same Chat and must not create cycles or branching.
4. `chatgroup_id` on a Chat cannot be changed after creation.
5. `ranking_formula` must be valid Python lambda syntax accepted by `asteval`.

## Open questions

- Should message deletion physically delete or soft-delete? Soft-delete preserves links in existing Events.
- Should `PUT /messages` accept partial updates (e.g., edit only `full_text`) or always require the full resource?
- Do we need a bulk message save endpoint for importing chat history?
