# Vistui Batch Consolidation System

## Context

Raw messages alone are not ideal long-term memory. The batch consolidation system processes older messages during idle periods and distills them into three structured memory objects:

- **Events** — summaries of contiguous message sequences within a Chat.
- **Topics** — summaries of recurring themes across non-contiguous messages in a ChatGroup.
- **Facts** — small, persistent facts about the user or world state in a ChatGroup.

The system is intentionally lazy: it does not need to run after every message. Modern chat context windows handle recent history; Vistui handles long-term memory.

Before any message can be consolidated, two prerequisite pipelines must complete:

- **Embedding pipeline** — computes the `embedding` vector.
- **Salience pipeline** — computes the `salience_score`.

These run for all messages, including the most recent protected messages. Events, Topics, and Facts that are rewritten by a consolidation pipeline have their `embedding` and `salience_score` cleared and later regenerated by the same embedding/salience batch jobs.

## Goals

1. Ensure every message has an embedding and a salience score.
2. Consolidate older messages into richer, more compact memory.
3. Run only during inactivity to avoid interfering with interactive chat.
4. Support pause/resume and graceful shutdown at almost any point.
5. Resume automatically after a server restart without losing work.

## Non-goals

- Real-time memory updates.
- Re-processing edited messages after they have been consolidated.
- Cross-ChatGroup consolidation.

## Timing and triggers

- The worker starts after `batch_inactivity_seconds` of no message activity (default 1 hour).
- Any new message pauses the worker gracefully: the current LLM sub-task is allowed to finish, but no new sub-task is started.
- A manual trigger endpoint is available for testing (`POST /admin/batch/trigger`).

## Worker design

- **Single worker.** Only one consolidation process runs at a time to avoid LLM rate limits and keep the design simple.
- **In-process scheduler.** APScheduler is embedded in the FastAPI application, using a Postgres `SQLAlchemyJobStore`.
- **Graceful stop.** The worker checks a `should_stop` flag between sub-tasks. If set, it stops before the next step.
- **Resume on restart.** Processing state is persisted in the database. On startup, the scheduler loads pending work and continues.

## Processing state machine

Each message has five independent processing flags in `processing_state`:

```json
{
  "embedding": "waiting | processing | done | failed",
  "salience": "waiting | processing | done | failed",
  "event": "waiting | processing | done | failed",
  "topic": "waiting | processing | done | failed",
  "fact": "waiting | processing | done | failed",
  "retry_count": 0,
  "last_failure_at": null
}
```

Rules:

- `embedding` and `salience` run for all messages, regardless of protected window.
- `event`, `topic`, and `fact` only run for messages outside the protected window (`protected_message_count` most recent messages in a Chat).
- The most recent messages are excluded from consolidation to avoid complications from edits, rerolls, and deletions.
- Older messages are processed first.
- Each pipeline can fail independently and is retried with exponential backoff.
- A processing step that exceeds `processing_timeout_seconds` is marked `failed`.
- A message is eligible for event/topic/fact consolidation only when both `embedding = done` and `salience = done`.

## Consolidation pipelines

### Prerequisite: embedding pipeline

**Scope:** per Message.

1. Select the oldest message with `embedding = waiting` or `embedding = null`.
2. Send `full_text` to the configured embedding provider.
3. Store the embedding vector.
4. Mark `embedding = done`.

If an Event, Topic, or Fact has `embedding = null`, it is also eligible for this pipeline.

### Prerequisite: salience pipeline

**Scope:** per Message.

1. Select the oldest message with `salience = waiting` or `salience_score = null`.
2. Send the message to the salience LLM prompt.
3. Parse and validate the score in `[0, 1]`.
4. Store the salience score.
5. Mark `salience = done`.

If an Event, Topic, or Fact has `salience_score = null`, it is also eligible for this pipeline.

### Event pipeline

**Scope:** per Chat.

1. Select the oldest unprocessed message (`event = waiting`) that is outside the protected window and has `embedding = done` and `salience = done`.
2. Load the message together with all subsequent unprocessed messages in the Chat until a natural break or until the current protected boundary.
3. Send the contiguous block to the event LLM prompt.
4. The LLM returns one or more events summarizing the block. A single message can belong to multiple events if the LLM decides the message spans event boundaries.
5. For each event:
   - If the event is new, create it.
   - If the event is the currently open (unfinished) event, update it in place.
   - Link the relevant messages to the event.
   - Clear the event's `embedding` and `salience_score` to `null` so they are regenerated by the prerequisite pipelines.
6. Mark the processed messages as `event = done`.

### Topic pipeline

**Scope:** per ChatGroup.

The topic pipeline uses two LLM calls.

#### Step A: identify candidate topics

1. Select the oldest unprocessed message (`topic = waiting`) outside the protected window, with `embedding = done` and `salience = done`.
2. Search existing Topics in the ChatGroup using the same retrieval ranking logic as interactive retrieval.
3. Send the message and the top matching topics to the `topic_candidate` LLM prompt.
4. The LLM selects up to `max_topics_per_message` existing topics that might need updating, or decides that one or more new topics should be created.

Outputs:

- List of existing topic IDs to update (0 to `max_topics_per_message`).
- Optional list of new topic texts to create.

#### Step B: update each candidate topic

1. For each selected existing topic, send the topic, the new message, and the ChatGroup context to the `topic_update` LLM prompt.
2. The LLM decides whether to update the topic or leave it unchanged.
3. If updated:
   - Rewrite the topic text in place.
   - Link the current message to the topic.
   - Clear the topic's `embedding` and `salience_score` to `null`.
4. If new topics were requested in Step A:
   - Create each Topic.
   - Link the current message to each new topic.
   - Leave `embedding` and `salience_score` as `null` for the prerequisite pipelines.
5. Mark the message as `topic = done`.

### Fact pipeline

**Scope:** per ChatGroup.

The fact pipeline uses two LLM calls.

#### Step A: identify candidate facts

1. Select the oldest unprocessed message (`fact = waiting`) outside the protected window, with `embedding = done` and `salience = done`.
2. Search existing Facts in the ChatGroup.
3. Send the message and the top matching facts to the `fact_candidate` LLM prompt.
4. The LLM selects up to `max_facts_per_message` existing facts that might need updating, or decides that a new fact should be created.

Outputs:

- List of existing fact IDs to update (0 to `max_facts_per_message`).
- Optional new fact text to create.

#### Step B: update each candidate fact

1. For each selected existing fact, load the message plus the last `fact_history_count` messages linked to that fact.
2. Send the bundle to the `fact_update` LLM prompt.
3. The LLM decides whether to update the fact or leave it unchanged.
4. If updated:
   - Rewrite the fact text in place.
   - Link the current message to the fact.
   - Clear the fact's `embedding` and `salience_score` to `null`.
5. If a new fact was requested in Step A:
   - Create the Fact.
   - Link the current message to it.
   - Leave `embedding` and `salience_score` as `null` for the prerequisite pipelines.
6. Mark the message as `fact = done`.

A fact is rewritten in place when the underlying truth changes (e.g., "Mom is sick" → "Mom is on vacation").

## Work scheduling

The worker operates in rounds:

1. Pick the oldest waiting message across all Chats and all ChatGroups, regardless of pipeline.
2. Within that message, run the pipelines in order:
   1. `embedding` and `salience` prerequisites first if not done.
   2. Then `event`, `topic`, `fact` consolidation in that order. Each step is a separate sub-task.
3. Messages are processed one by one in chronological order, not ChatGroup by ChatGroup.
4. Commit results and state changes after each sub-task.
5. Check `should_stop`. If true, exit the round.
6. Repeat.

This design ensures that:

- Work is fine-grained and resumable.
- A server restart loses at most the in-flight LLM call.
- A graceful stop can occur between any two sub-tasks.
- A message never enters event/topic/fact processing without embedding and salience.
- The worker is message-oriented, not ChatGroup-oriented.

## Idempotency and retries

- A sub-task is considered complete only after the database commit.
- If a sub-task fails or times out, the message's flag is set to `failed` and `retry_count` is incremented.
- Retries use exponential backoff with a maximum delay (e.g., 5 minutes).
- After a maximum number of retries, the message is marked `done` for that pipeline to avoid infinite loops; the failure is logged and exposed in admin status.

## Interaction with message edits

- If a message is edited after it has already been processed, its processing flags are **not** reset.
- This is an accepted limitation due to the difficulty of re-deriving downstream memory.
- The `protected_message_count` window reduces the chance of editing affecting unprocessed work.

## Failure handling

| Scenario | Behavior |
|----------|----------|
| LLM call fails | Mark sub-task `failed`, retry later. |
| LLM returns invalid JSON | Mark `failed`, retry later with structured-repair note. |
| Embedding fails after LLM succeeds | Save the consolidated object without embedding, mark `failed`, retry embedding later. |
| Server restarts mid-task | In-flight task is lost; state remains `processing` or `failed` and is retried on startup. |
| New message arrives | Worker finishes current sub-task, then pauses. |

## Admin observability

Two endpoints provide insight:

```
POST /admin/batch/trigger
GET  /admin/batch/status
```

Status response:

```json
{
  "running": true,
  "current_chatgroup_id": "uuid-or-null",
  "current_pipeline": "fact",
  "current_message_id": "uuid-or-null",
  "queue_depth": 42,
  "recent_failures": 3
}
```

## Performance and cost considerations

- Serial processing avoids rate limits but limits throughput.
- LLM calls dominate cost and latency.
- Embeddings and salience scores for consolidated objects are computed lazily by the prerequisite pipelines.
- The protected window reduces churn and wasted LLM calls from edits.

## Open questions

- Should the event pipeline be scoped to Chat or to ChatGroup?
- Should failed pipelines send alerts or only appear in logs?
- Should there be a reaper job to clean up dangling links from deleted messages?
